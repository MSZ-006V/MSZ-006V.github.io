# Redis 数据库笔记(Redis Notes)
## Introduction
- Redis 是一种基于内存的数据库，对数据的读写操作都是在内存中完成，因此**读写速度非常快**，常用于缓存，消息队列、分布式锁等场景。
- Redis 提供了多种数据类型来文持不同的业务场景，比如 String(字符串)、Hash(哈希)、 List (列表)、Set(集合)、Zset(有序集合)、Bitmaps(位图)、HyperLogLog(基数统计)、GEO(地理信息)、Stream(流)，并且对数据类型的操作都是原子性的，因为执行命令由单线程负责的，不存在并发竞争的问题。
- 除此之外，Redis 还支持事务 、持久化、Lua 脚本、多种集群方案(主从复制模式、哨兵模式、切片机群模式)、发布/订阅模式，内存淘汰机制、过期删除机制等等。
## 数据类型
- 常见数据类型：string，hash, list, set, zest(有序集合), bitmap, hyperloglog, geo, stream
### string
- key-value结构，key可以是字符串，数字，value最大长度512M
- 底层实现是int和SDS（动态字符串），SDS可以保存文本数据/二进制数据，SDS获取长度时间复杂度是O(1)，拼接SDS若超出长度会自动扩容，不会溢出
```sql
SET Key1 V1 -- 增加key-value类型
GET Key1
EXISTS Key1 -- 判断是否存在，返回integer
STRLEN Key1 -- 返回键值长度
DEL Key1 -- 删除
MSET key1 v1 key2 v2 -- 批量设置
MGET Key1 Key2 -- 批量获取
INCR number, INCRBY number 5, DECR number, DECRBY number 5 -- 自增1，自增5，自减
```
- 场景：共享session信息，分布式锁，计数器，常规对象
### list
- 字符串链表，按照插入顺序排序，可以向头或尾添加元素
- 最大长度2^32-1
- 底层实现是双向链表或压缩链表（redis3.2后，由quicklist实现）
```sql
LPUSH key value [value ...] -- 将一个或多个值value插入到key列表的表头(最左边), 最后的值在最前面
RPUSH key value [value ...] -- 将一个或多个值value插入到key列表的表尾(最右边)
LPOP key
RPOP key
BLPOP key [key ...] timeout -- 从key列表表头弹出一个元素，没有就阻塞timeout秒，如果timeout=0则一直阻塞
BRPOP key [key ...] timeout -- 从key列表表尾弹出一个元素，没有就阻塞timeout秒，如果timeout=0则一直阻塞
```
- 场景：消息队列（可以使用BRPOP阻塞式读取，没读取到队列消息就自动阻塞，直到有新的消息写入再开始读取新的）
### hash
- 一个键值对key-value的集合，value形式是value= [{field1, value1}, ...{fieldN, valueN}],（string则是一个value对应一个key）
- 底层为哈希表或压缩链表实现（redis7.0后由压缩链表替代为listpack实现）
```sql
HSET key field value -- 存储一个哈希表key的键值
HGET key field -- 获取哈希表key对应的field键值
HMSET key field value [field value...] -- 在一个哈希表key中存储多个键值对
HMGET key field [field ...] -- 批量获取哈希表key中多个field键值
HDEL key field [field ...] -- 删除哈希表key中的field键值
HLEN key -- 返回哈希表key中field的数量
HGETALL key -- 返回哈希表key中所有的键值
HINCRBY key field n -- 为哈希表key中field键的值加上增量n
```
- 场景：缓存对象
### set
- 无序，唯一的键值集合，存储顺序不会按照插入的先后顺序存储
- 一个集合存储2^32-1哥元素，支持增删改查，并集交集差集
- list可以储存重复元素，set不行；list按照插入先后顺序存储，set无序
- 底层为哈希表或整数集合实现
```sql
SADD key member [member ...] -- 往集合key中存入元素，元素存在则忽略，若key不存在则新建
SREM key member [member] -- 从集合key中删除元素
SMEMBERS key -- 获取集合key中所有元素
SCARD key -- 获取集合key中的元素个数
SISMEMBER key member -- 判断member元素是否存在于集合key中
SRANDMEMBER key [count] -- 从集合key中随机选出count个元素，元素不从key中删除
SPOP key [count] -- #从集合key中随机选出count个元素，元素从key中删除
-- SINTER SINTERSTORE, SUNION SUNIONSTORE, SDIFF SDIFFSTORE. 交集，并集，差集运算
```
- 场景：（交并差开销巨大，在主库中执行容易阻塞，建议将数据返回客户端后由客户端执行）点赞功能（一个用户只能点一次），共同关注（交集），抽奖（有去重，SRANDMEMBER随机抽取元素或SPOP）
## 持久化技术Redis Persistence
- 指将程序运行过程中产生的数据永久存储到非易失性存储介质（如硬盘、SSD）中的技术。其目的是在系统重启或发生故障后依然能够恢复数据。持久化技术广泛用于数据库、缓存系统和日志管理中，确保数据的长期保存和一致性
- Redis默认RDB快照
### RDB快照持久化Redis Database Backup
- RDB持久化是通过生成内存数据的快照并保存到磁盘中来实现的。Redis会在指定的时间间隔内自动触发快照操作，将当时的数据存储到RDB文件中，这样可以在Redis重启时恢复数据。RDB的特点是节省磁盘空间，写入效率较高，适合用于备份和灾难恢复，但可能会丢失最后一次快照后的数据。RDB写入的是二进制数据
- 优点：体积小，适合冷备份，恢复数据快
- 缺点：上一次快照到下一次快照之间的操作不会被保存，备份频率设置过低会导致数据不实时，过高则影响性能
- `save`命令（在主线程生成RDB文件，可能会阻塞主线程）和`bgsave`（创建子线程来生成RDB文件，避免阻塞），同时是全量快照（每次执行快照，把内存中所有数据记录到磁盘中）
    - 创建的子进程会复制父进程的页表，指向同一块物理内存
- 执行`bgsave`中，主线程依然可以继续处理操作命令，同时数据也可以修改————写时复制技术（Copy on Write, COW）
    - 主线程要修改数据的物理内存先复制一份(键值对'A')，然后主线程在这个数据副本(键值对 'A')进行修改操作。与此同时，bgsave子进程可以继续把原来的数据(键值对'A')写入到RDB文件
    - 发生写时复制后，`bgsave`保存的数据仍然是未修改的
### AOF 追加日志持久化 Append Only File
- 通过记录每个写入操作的日志并追加到文件中来实现的，Redis会将每个写命令追加写入到AOF文件中。可以根据需要选择不同的同步策略（每秒同步、每次写操作同步等）。AOF可以最大程度减少数据丢失，适合对数据一致性要求较高的场景。AOF写入的是命令
- 优点：持久化更加实时，数据丢失较少；AOF文件是可读的文本日志，便于分析和修改
- 缺点：文件体积较大，AOF文件会不断增大，需要定期重写和压缩；写操作影响影响性能
    - AOF文件到达一定体积，触发重写机制：在重写时，读取当前数据库中的所有键值对，然后将每一个键值对用一条命令记录到新的AOF文件，等到全部记录完后，就将新的AOF文件替换掉现有的AOF文件
- 执行流程：
    - Redis执行完写操作命令后，会将命令追加到server.aof_buf缓冲区
    - 通过write()系统调用，将aof_buf缓冲区的数据写入到AOF文件，此时数据并没有写入到硬盘而是拷贝到了内核缓冲区page cache，等待内核将数据写入硬盘
    - 具体内核缓冲区的数据什么时候写入到硬盘，由内核决定
- 三种AOF写回策略
    - Always，每次写操作命令执行完后，同步将AOF日志数据写回硬盘 - 高可靠
    - Everysec，每次写操作命令执行完后，先将命令写入到AOF文件的内核缓冲区，然后每隔一秒将缓冲区里的内容写回到硬盘 - 折中，可能丢失一部分数据
    - No，每次写操作命令执行完后，先将命令写入到AOF文件的内核缓冲区，再由操作系统决定何时将缓冲区内容写回硬盘 - 高性能

## 过期删除策略
- Redis可以设置对过期的key进行删除，支持设置key的TTL——即过期键值删除策略。
- 当设置key的过期时间时，redis会将key带上过去时间写入过期字典(expires dict)，里面保存了所以key的过期时间
- Expires dict（哈希表）中数据结构有key（指针，指向某个对象）和value（long long类型，过期时间）
- 三种：
    - 定时删除：设置key过期时间时，同时设置定时时间，到达时间时由事件处理器执行key删除操作（对内存友好，但是会占用CPU时间）
   - 惰性删除：不主动删除，当有访问这个key的时候检查是否过期，是则删除（内存不友好，CPU友好，不会占用很多时间）
    - 定期删除：一段时间随机取出部分key检查是否过期（折中，但难以确定检查频率和检查数量）
- redis选择惰性删除与定期删除策略结合
## 内存淘汰策略
- redis运行内存超过设定最大内存时，删除部分key
- 内存淘汰策略：
    - 不进行数据淘汰：
        - noeviction(Redis3.0之后，默认淘汰策略): 它表示当运行内存超过最大设置内存时，不淘汰任何数据，这时如果有新的数据写入，会报错通知禁止写入，不淘汰任何数据，但是如果没有数据写入的话，只是单纯的查询或者删除操作的话，还是可以正常工作。
    - 进行数据淘汰：
        - 在设置了过期时间的数据中进行淘汰:
            - volatile-random: 随机淘汰设置了过期时间的任意键值;
            - volatile-ttl: 优先淘汰更早过期的键值。
            - volatile-lru(Redis3.0之前，默认的内存淘汰策略): 淘汰所有设置了过期时间的键值中，最久未使用的键值;
            - volatile-lfu(Redis 4.0 后新增的内存淘汰策略): 淘汰所有设置了过期时间的键值中，最少使用的键值;
        - 在所有数据范围内进行淘汰:
            - allkeys-random: 随机淘汰任意键值;
            - allkeys-lru: 淘汰整个键值中最久未使用的键值,
            - allkeys-lfu(Redis 4.0 后新增的内存淘汰策略): 淘汰整个键值中最少使用的键值。
## LFU和LRU
- LFU（least frequently use）：最近最不常用，根据访问次数来淘汰
- LRU（least recetly use）：最近最少使用，根据最后访问的时间来淘汰
